# Stage 0.5 – Planning the Planning

Meta-doc: how we'll plan and build this weekend hack, and the shape of the work ahead.

## Project in one line

A frontend-only, scrubbable map/globe visualiser that animates the timeline of rocket launches by country using real data.

Seed: [`readme.md`](../readme.md).

## Ways of working

- **Weekend-hack scope.** Success = one demoable artefact by end of weekend, not a platform.
- **Pragmatic over perfect.** Pick boring, well-understood defaults. Avoid decisions that paint us into a corner for a future v2, but don't over-engineer for it either.
- **Easy to return to.** Future-me should be able to `open index.html` in a browser months later and have it Just Work. No stale toolchain, no `npm install` bit-rot.
- **Low/no build.** Preact + htm loaded via ES modules + import maps in the browser. No bundler, no transpiler, no package.json unless strictly needed.
- **One folder, plain text.** Code and docs live side-by-side in this repo. Docs are markdown; code is ES modules.
- **Real data only** for launches (no fabricated numbers), but synthesised/averaged playback is fine when the source data is annual.

## Key questions to resolve in Stage 1

Product:

- What's the smallest demoable slice? (Working hypothesis: globe + ~10 top launching countries + draggable scrubber + one animation per launch.)
- Globe (3D) or map (2D)? Trade-off: globe feels right for "space" but is more rendering work.
- Per-launch events vs annual-average playback vs both?
- Date range — start of space era (Oct 1957) to today?
- Country granularity — launching nation, launch-site nation, or both?
- Is launch metadata (rocket, payload, success/failure) needed for MVP or v2?
- Playback controls — scrub, play/pause, speed, jump-to-year?
- Mobile required for MVP, or desktop-only?

Production:

- Hosting target — GitHub Pages, Netlify drop, or just `file://`?
- Data licensing / attribution — what must be visible in-app?
- Offline behaviour — ship the dataset as a static JSON in the repo?

## Decisions already locked in

- Frontend-only, plain JavaScript.
- Preact + htm, no build step.
- ES modules via import map.
- Single-page app, likely single `index.html`.
- Real launch data, statically bundled in the repo.

## Out of scope (explicit non-goals for MVP)

- Backend, API, user accounts.
- Live/streaming launch data.
- Editing or contributing data in-app.
- Deep per-launch detail pages.
- i18n.
